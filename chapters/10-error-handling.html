Chapter 10: Error Handling

Learn to anticipate, catch, and recover from errors so your programs never crash unexpectedly

The Idea

Expect the Unexpected

Every programmer makes mistakes. The difference between a beginner and an expert? Experts plan for errors.

Think about it: what happens if a user types their name when your program is asking for a number? What if you try to open a file that doesn’t exist? What if your code tries to divide by zero? Without error handling, your program would simply crash — it stops running and shows an angry red message. That’s a terrible experience for the person using your program.

Errors aren’t failures — they’re information. When something goes wrong, Python tells you exactly what happened and where it happened. Once you learn to read error messages, they become your best friend for debugging. And once you learn to handle errors, you can write programs that recover gracefully instead of crashing.

You see error handling everywhere in the real world. When a website says “Are you sure you want to delete this?”, that’s handling a potential mistake before it happens. When a form highlights a field in red and says “Please enter a valid email address”, that’s catching bad input. When your phone says “No internet connection — try again later” instead of just freezing, that’s graceful error handling. Professional software is full of it.

By the end of this chapter, you will be able to:

  • Understand the most common types of Python errors and what causes them
  • Use try and except to catch errors and keep your program running
  • Handle specific error types differently depending on what went wrong
  • Use the finally block to run clean-up code no matter what happens

Example 2: Try/Except to the Rescue

Now for the good part — we can catch errors before they crash our program using try and except. You put the “risky” code inside a try block, and if an error occurs, Python jumps to the except block instead of crashing.

Python Code 
# Without try/except, this would crash!
try:
    result = 10 / 0
except ZeroDivisionError:
    print("Can't divide by zero!")

print("Program keeps running!")
(click "Run Code" to see output)

Notice how the program doesn’t crash. When Python hits the error inside the try block, it jumps straight to the except block and runs that code instead. Then it carries on with the rest of the program as normal. The line "Program keeps running!" still prints — without try/except, it never would.

Python Code 
# A practical example: safely converting user input
user_input = "hello"  # Imagine the user typed this

try:
    number = int(user_input)
    print("You entered:", number)
except ValueError:
    print("That's not a valid number!")
    print("You typed:", user_input)
(click "Run Code" to see output)
Try This: Change user_input to "42" and run the code again. What happens when the conversion succeeds? The except block is simply skipped. Try other values like "3.14" or "" (an empty string) to see what happens.

Example 3: Multiple Except Blocks and Finally

Sometimes different things can go wrong in the same piece of code. You can use multiple except blocks to handle each type of error differently. You can also add a finally block that always runs, whether an error occurred or not.

Python Code 
# Handling different errors differently
def divide(a, b):
    try:
        result = a / b
        print("Result:", result)
    except ZeroDivisionError:
        print("Error: You can't divide by zero!")
    except TypeError:
        print("Error: Both values must be numbers!")
    finally:
        print("--- Calculation attempt finished ---")

# Test with different inputs
print("Test 1: Normal division")
divide(10, 3)

print()
print("Test 2: Division by zero")
divide(10, 0)

print()
print("Test 3: Wrong type")
divide("ten", 2)
(click "Run Code" to see output)

Look at the output carefully. The finally block runs every single time — after a successful division, after a ZeroDivisionError, and after a TypeError. This makes finally perfect for clean-up tasks that must happen no matter what, like closing a file or saving progress.

Understand

How Does It Work?

Types of Errors

Python has many built-in error types (officially called exceptions). Here are the ones you’ll encounter most often:

Error Type What Causes It Example
SyntaxError Your code breaks Python’s grammar rules print("hello" — missing closing bracket
NameError You used a variable that doesn’t exist print(score) when score was never created
TypeError You used the wrong type of data "hello" + 5 — can’t add a string and an integer
ValueError Right type, but the value doesn’t make sense int("abc") — can’t convert letters to a number
ZeroDivisionError You tried to divide by zero 10 / 0
IndexError You tried to access a list index that doesn’t exist my_list[99] when the list only has 3 items
KeyError You tried to access a dictionary key that doesn’t exist my_dict["age"] when there is no "age" key

Important note: SyntaxError is different from the others. It means your code is incorrectly written and Python can’t even start running it. You cannot catch a SyntaxError with try/except because the error happens before your code runs. The only fix for a SyntaxError is to correct the code itself.

The Try/Except Structure

Here is the general pattern for handling errors in Python:

Python Code 
# The basic structure
try:
    # Code that might cause an error
    # (the "risky" code goes here)
    pass
except SomeErrorType:
    # What to do if that specific error happens
    pass
except AnotherErrorType:
    # What to do if a different error happens
    pass
finally:
    # Code that ALWAYS runs, error or not
    # (optional - you don't always need this)
    pass

Here’s how Python executes this:

  1. Python tries to run the code inside the try block
  2. If no error occurs, it skips all the except blocks and runs the finally block (if there is one)
  3. If an error occurs, Python stops the try block immediately, finds the matching except block, runs it, then runs the finally block
  4. If an error occurs but no matching except is found, the program crashes as normal (the finally block still runs first)

Catching Specific Errors vs General Except

You can write a “catch-all” except block that catches any error:

Python Code 
# Catching ANY error (use with caution!)
try:
    result = 10 / 0
except:
    print("Something went wrong!")

# Better: catch the specific error
try:
    result = 10 / 0
except ZeroDivisionError:
    print("Can't divide by zero!")
(click "Run Code" to see output)
Key Concept: Always Catch Specific Errors When Possible A bare except: (without naming an error type) will catch every error, including ones you didn’t expect. This sounds helpful, but it’s actually dangerous because it can hide bugs in your code. If you catch everything, you might never notice that your code has a logic error. Always name the specific error types you expect, like except ValueError: or except ZeroDivisionError:. This way, unexpected errors still show up and you can fix them.

The Finally Block

The finally block is optional, but when you use it, it always runs — whether the try block succeeded, whether an error was caught, or even if an error was not caught. This makes it ideal for clean-up tasks:

  • Closing a file that was opened
  • Saving data before the program ends
  • Printing a “finished” message
  • Resetting a value back to its default

Common Misconceptions

Watch out: Some learners think that wrapping everything in try/except is always a good idea. It isn’t. Using try/except wisely means catching errors you expect and know how to recover from — like a user typing letters when you need a number. Don’t catch everything blindly, because you’ll hide real bugs. If your code has a NameError because you misspelt a variable, you want to see that error so you can fix it — not sweep it under the carpet!
Watch out: Remember that SyntaxError cannot be caught with try/except. If Python can’t understand your code at all, it never gets as far as running the try block. The fix for a SyntaxError is always to correct the code itself — check for missing colons, brackets, or quotation marks.
